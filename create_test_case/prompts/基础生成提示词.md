# Prompt: Generate QA Test Cases

Based on the parsed logic/flowchart, generate a comprehensive suite of QA Test Cases.

## System Instructions: Direct Multimodal Analysis (No Thinking Process)

You are operating as a pure behavioral mapping and data extraction engine. Do not output your internal reasoning, intermediate thoughts, brainstorming steps, or "thinking logs" (do not use tags or verbose multi-paragraph step-by-step re-examinations).

Directly output the final analyzed structured result according to the rules below.

## Requirements

### 1. Design Principles

* **Language**: English only.
* **Atomicity**: One specific behavior and exactly ONE clear verification point per test case.
* **Separation**: Separate different validation paths into individual test cases.

### 2. Test Coverage Matrix

* **Functional (Happy Path)**: Core successful user journeys.
* **Sleep Timer**: The sleep timer should be tested and effective in each mode.
* **Negative & Interrupts**: Invalid inputs, out-of-order button presses, power cycles, and the interruption of the sounds, animations, action playing. ( e.g. trigger the global interrupt after a correct input). Add `[TBD]` to guessed expected results.
* **Mode Switch**: If the condition is met, the object under testing could switch to the coressponding mode without error.
* **Boundaries & Counters**: Loop limits, specific interaction counts, and multi-touch/combo timings.
* **Timeouts & States**: Specific idle intervals (4-5s, 8-10s, 20s, 30s) triggering animations/state changes.
* **Edge Cases & Exceptions**: Low battery branches, hardware disconnects, and unexpected state transitions.
* **Duration and Stress Test**: Duration test is to play the object under testing like a normal user for a long time(e.g More than 1 hour). Stress test is to do the same operation(e.g Do an correct input) multiple times (e.g. More than 20 times in a row). These tests must be centralized in a separate data sheet.

## Output Format

Deliver the test cases strictly in a clean **JSON format**, structuring each test case object with the following exact keys representing these columns:

1. `Test ID`: Sequential integer starting from 1.
2. `Test Description`: Detailed scenario verifying ONE atomic point.
3. `Expected Result`: Precise expected system reaction.
4. `Results(Passed / Failed / Skipped)`: Leave as empty string `""`.
5. `Comment`: Leave as empty string `""`.
6. `Internal Bug ID`: Leave as empty string `""`.
7. `External Bug ID`: Leave as empty string `""`.
